        function syncFiltersToURL() {
            const params = new URLSearchParams();
            if (activeFilter !== 'All') params.set('s', activeFilter);
            if (activeType !== 'All') params.set('t', activeType);
            const str = params.toString();
            history.replaceState(null, '', str ? '?' + str : location.pathname + location.hash);
        }

                    <p>Storage slot collisions and layout corruption are silent killers in upgradeable contracts.</p>

                <article class="vuln-card" id="storage-collision" data-severity="High" data-type="Infrastructure">
                    <div class="vuln-header">
                        <div class="vuln-title-wrap">
                            <h3 class="vuln-title">Storage Collision & Layout Corruption</h3>
                            <p class="vuln-subtitle">Variables overwrite each other across upgrades or proxy boundaries</p>
                        </div>
                        <span class="severity-badge severity-high">High</span>
                    </div>
                    <p>Upgradeable contracts share storage between the proxy and implementation. Adding, removing, or reordering state variables in an upgrade corrupts all existing data. Even inheritance order changes can shift slot assignments. ERC-7201 namespaced storage eliminates this class of bug entirely.</p>

                    <div class="vuln-details">
                        <div class="detail-item">
                            <h4>Root Cause</h4>
                            <p>Solidity assigns storage slots sequentially (slot 0, 1, 2…). Any change to variable order or count between upgrades misaligns slots in the live proxy. With <code>delegatecall</code>, proxy admin variables can also collide with implementation storage at the same slots.</p>
                        </div>
                        <div class="detail-item">
                            <h4>Unstructured Storage (Pre-ERC-7201)</h4>
                            <p>OpenZeppelin's original pattern used a manually chosen high slot via <code>keccak256("eip1967.proxy.implementation") - 1</code>. Collision-resistant but requires discipline — one wrong slot and data is overwritten silently.</p>
                        </div>
                    </div>

                    <div class="code-example">
                        <div class="code-block">
                            <span class="code-label vulnerable">❌ VULNERABLE — storage layout corruption on upgrade</span>
                            <pre><code class="language-solidity">// V1
contract TokenV1 {
    address public owner;   // slot 0
    uint256 public supply;  // slot 1
}

// V2 — BAD: inserted variable shifts slots
contract TokenV2 {
    address public owner;   // slot 0
    uint256 public paused;  // slot 1  ← NEW: supply now reads garbage
    uint256 public supply;  // slot 2  ← was slot 1!
}

// Any read of `supply` after upgrade returns corrupted data</code></pre>
                        </div>
                        <div class="code-block">
                            <span class="code-label safe">✅ SAFE — ERC-7201 namespaced storage</span>
                            <pre><code class="language-solidity">// ERC-7201: keccak256(abi.encode(uint256(keccak256("myapp.token")) - 1)) & ~bytes32(uint256(0xff))
// Computed once off-chain, never changes between upgrades

/// @custom:storage-location erc7201:myapp.token
contract TokenStorageV1 {
    struct Storage {
        address owner;
        uint256 supply;
    }

    // Deterministic slot — immune to layout changes
    bytes32 private constant STORAGE_SLOT =
        0x1234abcd...;  // ERC-7201 derived slot

    function _getStorage() private pure returns (Storage storage s) {
        assembly { s.slot := STORAGE_SLOT }
    }

    function getOwner() public view returns (address) {
        return _getStorage().owner;
    }
}

// V2: add fields to Storage struct — existing slots are never affected
struct Storage {
    address owner;
    uint256 supply;
    bool paused;   // ← appended safely, no collision
}</code></pre>
                        </div>
                    </div>

                    <div class="mitigation">
                        <h4>Mitigations</h4>
                        <ul>
                            <li>Use <strong>ERC-7201 namespaced storage</strong> for all upgradeable contracts — compile with <code>@custom:storage-location</code> annotation</li>
                            <li>Never remove or reorder variables in a storage struct — only append</li>

                        </div>

                    </div>
                    <p>The Curve Finance exploit (August 2023, ~$73M) was caused by a reentrancy guard failure in Vyper versions <code>0.2.15</code>, <code>0.3.0</code>, and <code>0.3.1</code>.                         </div>

                            <h4>Bug 2 — Storage Array Bounds Check Bypass (pre-0.3.10)</h4>

                        </div>

                            <h4>Trap — All Functions Are External by Default</h4>
                            <p>Unlike Solidity, Vyper functions are <code>external</code> by default (not <code>internal</code>). Internal calls use <code>self.method()</code>. Misunderstanding this can expose functions intended to be internal-only.</p>
                        </div>
