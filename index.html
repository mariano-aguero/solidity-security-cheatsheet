01

Reentrancy Attack

External calls allow recursive exploitation

Critical

An attacker exploits external calls that callback into the vulnerable contract before state updates complete. This enables draining funds through recursive calls.

Real-world Impact: The DAO hack (2016) — $60M stolen using this exact vulnerability.

Types of Reentrancy:

  • Single-function: Attacking the same function recursively
  • Cross-function: Attacking different functions that share state
  • Cross-contract: Exploiting reentrancy across multiple contracts
  • Read-only: Exploiting view functions that return stale state during an active external call — no state is modified but stale data is consumed downstream
❌ READ-ONLY REENTRANCY (Curve/Vyper style) 
// Pool sends ETH back BEFORE burning LP tokens
// → virtual_price() still reflects pre-burn inflated value
contract VulnerablePool {
    uint public totalSupply;

    // view function returns STALE inflated value during reentrancy
    function get_virtual_price() external view returns (uint) {
        return address(this).balance * 1e18 / totalSupply;
    }

    function remove_liquidity(uint lpAmount) external {
        uint ethOut = lpAmount * address(this).balance / totalSupply;

        // ETH sent here — attacker's receive() fires
        // At this point totalSupply NOT yet decreased!
        (bool ok,) = msg.sender.call{value: ethOut}("");
        require(ok);

        totalSupply -= lpAmount; // Too late — attacker already read stale price
    }
}

// Attacker exploits a lending protocol that uses VulnerablePool.get_virtual_price()
contract ReadOnlyAttacker {
    VulnerablePool pool;
    ILendingProtocol lending; // uses pool.get_virtual_price() as collateral oracle

    receive() external payable {
        // During remove_liquidity, before totalSupply decreases:
        // virtual_price() is inflated → collateral value is overstated
        uint inflatedCollateral = lending.getCollateralValue(address(pool));
        lending.borrow(inflatedCollateral); // Borrow more than collateral is worth!
    }
}

// FIX: add reentrancy lock to view functions in DeFi pools
uint private _locked = 1;
modifier nonReentrantView() {
    require(_locked == 1, "Reentrant call");
    _;
}
❌ VULNERABLE 
contract VulnerableBank {
    mapping(address => uint) public balances;

    function withdraw(uint amount) external {
        require(balances[msg.sender] >= amount, "Insufficient");

        // External call BEFORE state update - DANGEROUS!
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success, "Transfer failed");

        // Attacker's receive() calls withdraw() again
        // before this line executes
        balances[msg.sender] -= amount;
    }
}

// Attacker Contract
contract Attacker {
    VulnerableBank public bank;

    function attack() external payable {
        bank.deposit{value: msg.value}();
        bank.withdraw(msg.value);
    }

    receive() external payable {
        if (address(bank).balance >= 1 ether) {
            bank.withdraw(1 ether); // Re-enter!
        }
    }
}
✅ SECURE - Checks-Effects-Interactions 
contract SecureBank {
    mapping(address => uint) public balances;

    function withdraw(uint amount) external {
        // 1. CHECKS
        require(balances[msg.sender] >= amount, "Insufficient");

        // 2. EFFECTS - Update state BEFORE external call
        balances[msg.sender] -= amount;

        // 3. INTERACTIONS - External call LAST
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success, "Transfer failed");
    }
}
✅ SECURE - Using ReentrancyGuard 
import "@openzeppelin/contracts/utils/ReentrancyGuard.sol";

contract SecureBankWithGuard is ReentrancyGuard {
    mapping(address => uint) public balances;

    function withdraw(uint amount) external nonReentrant {
        require(balances[msg.sender] >= amount);
        balances[msg.sender] -= amount;
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success);
    }

    function transfer(address to, uint amount) external nonReentrant {
        require(balances[msg.sender] >= amount);
        balances[msg.sender] -= amount;
        balances[to] += amount;
    }
}

🛡️ Prevention

  • Always use Checks-Effects-Interactions pattern
  • Use ReentrancyGuard for all state-changing functions
  • Consider nonReentrant on view functions in DeFi protocols
  • Be aware of cross-contract reentrancy in multi-contract systems
02

Integer Overflow/Underflow

Arithmetic operations wrap around silently

Critical

In Solidity < 0.8.0, arithmetic operations that exceed type bounds wrap around to unexpected values, enabling attackers to mint unlimited tokens or bypass balance checks.

Real-world Impact: BEC Token (2018) — Unlimited token minting via overflow.
❌ VULNERABLE (Solidity < 0.8.0) 
contract VulnerableToken {
    mapping(address => uint256) public balances;

    // Overflow: receivers.length * value can wrap to small number
    function batchTransfer(address[] memory receivers, uint256 value) public {
        uint256 total = receivers.length * value; // Can overflow!
        require(balances[msg.sender] >= total);

        balances[msg.sender] -= total;
        for (uint i = 0; i < receivers.length; i++) {
            balances[receivers[i]] += value;
        }
    }

    // Underflow: 0 - 1 = MAX_UINT256
    function withdraw(uint256 amount) public {
        balances[msg.sender] -= amount; // If balance is 0, becomes huge!
        payable(msg.sender).transfer(amount);
    }
}

// Example: uint8 max = 255
// 255 + 1 = 0 (overflow)
// 0 - 1 = 255 (underflow)
✅ SECURE - Solidity >= 0.8.0 
// Solidity >= 0.8.0 has automatic overflow checks
contract SecureToken {
    mapping(address => uint256) public balances;

    function batchTransfer(address[] memory receivers, uint256 value) public {
        uint256 total = receivers.length * value; // Reverts on overflow
        require(balances[msg.sender] >= total);

        balances[msg.sender] -= total; // Reverts on underflow
        for (uint i = 0; i < receivers.length; i++) {
            balances[receivers[i]] += value;
        }
    }
}

// For legacy code, use SafeMath:
import "@openzeppelin/contracts/utils/math/SafeMath.sol";

contract LegacySecure {
    using SafeMath for uint256;

    function transfer(address to, uint256 amount) public {
        balances[msg.sender] = balances[msg.sender].sub(amount);
        balances[to] = balances[to].add(amount);
    }
}
⚠️ CAREFUL - Unchecked blocks 
// Only use unchecked when mathematically proven safe
contract OptimizedLoop {
    function sum(uint256[] calldata arr) external pure returns (uint256 total) {
        uint256 length = arr.length;
        for (uint256 i = 0; i < length;) {
            total += arr[i];
            unchecked { ++i; } // Safe: can't overflow before gas limit
        }
    }

    // ❌ DANGEROUS
    function unsafeDecrement(uint256 x) external pure returns (uint256) {
        unchecked { return x - 1; } // Can underflow if x == 0!
    }
}

🛡️ Prevention

  • Use Solidity >= 0.8.0 for automatic overflow checks
  • Use SafeMath library for older versions
  • Only use unchecked when mathematically proven safe
  • Be extra careful with multiplication before division
03

Access Control Vulnerabilities

Missing or incorrect permission checks

Critical

Missing or incorrect access controls allow unauthorized users to execute privileged functions, potentially taking over the entire contract.

Real-world Impact: Parity Multisig Wallet (2017) — $30M stolen, $150M frozen.
❌ VULNERABLE 
contract VulnerableWallet {
    address public owner;

    constructor() { owner = msg.sender; }

    // Anyone can call this!
    function setOwner(address _newOwner) public {
        owner = _newOwner;
    }

    // Anyone can withdraw!
    function withdraw() public {
        payable(msg.sender).transfer(address(this).balance);
    }
}

// Unprotected initializer (proxy pattern)
contract VulnerableImplementation {
    address public owner;
    bool public initialized;

    // Anyone can call initialize on the implementation!
    function initialize(address _owner) external {
        require(!initialized);
        owner = _owner;
        initialized = true;
    }
}
✅ SECURE - Ownable2Step (Recommended) 
// ❌ Ownable: one-step transfer — typo = permanent ownership loss
import "@openzeppelin/contracts/access/Ownable.sol";
contract Risky is Ownable {
    constructor() Ownable(msg.sender) {}
    // transferOwnership(wrongAddress) → contract is orphaned forever
}

// ✅ Ownable2Step: new owner must call acceptOwnership()
import "@openzeppelin/contracts/access/Ownable2Step.sol";

contract SecureOwnable is Ownable2Step {
    constructor() Ownable(msg.sender) {}

    function sensitiveAction() external onlyOwner {
        // Only owner can call
    }
}
// Flow: owner calls transferOwnership(newOwner)
//       newOwner calls acceptOwnership() — atomic, no typo risk
✅ SECURE - Role-based Access 
import "@openzeppelin/contracts/access/AccessControl.sol";

contract SecureProtocol is AccessControl {
    bytes32 public constant ADMIN_ROLE = keccak256("ADMIN_ROLE");
    bytes32 public constant OPERATOR_ROLE = keccak256("OPERATOR_ROLE");

    constructor() {
        _grantRole(DEFAULT_ADMIN_ROLE, msg.sender);
        _grantRole(ADMIN_ROLE, msg.sender);
    }

    function adminFunction() external onlyRole(ADMIN_ROLE) {
        // Only admins can call
    }

    function operatorFunction() external onlyRole(OPERATOR_ROLE) {
        // Only operators can call
    }
}

// Protected initializer for proxies
import "@openzeppelin/contracts-upgradeable/proxy/utils/Initializable.sol";

contract SecureImplementation is Initializable {
    address public owner;

    /// @custom:oz-upgrades-unsafe-allow constructor
    constructor() { _disableInitializers(); }

    function initialize(address _owner) external initializer {
        owner = _owner;
    }
}

🛡️ Prevention

  • Use Ownable2Step instead of Ownable — prevents accidental ownership loss via typo
  • Use OpenZeppelin's AccessControl for fine-grained roles
  • Protect initializers in upgradeable contracts with initializer modifier
  • Use _disableInitializers() in implementation constructors
  • Review function visibility carefully (public vs external vs internal)
  • Never leave DEFAULT_ADMIN_ROLE without a secure multi-sig

tx.origin Authentication

Phishing attacks via contract intermediary

High

Using tx.origin for authentication allows phishing attacks where a malicious contract tricks users into executing transactions that drain their wallets.

❌ VULNERABLE 
contract VulnerableWallet {
    address public owner;

    function transfer(address to, uint amount) public {
        // tx.origin is the EOA that started the transaction
        require(tx.origin == owner, "Not owner");
        payable(to).transfer(amount);
    }
}

// Attacker contract
contract PhishingAttack {
    VulnerableWallet wallet;
    address attacker;

    constructor(address _wallet) {
        wallet = VulnerableWallet(_wallet);
        attacker = msg.sender;
    }

    // Victim calls this thinking it's legitimate
    function claimReward() external {
        // tx.origin is still the victim!
        wallet.transfer(attacker, address(wallet).balance);
    }
}
✅ SECURE - Use msg.sender 
contract SecureWallet {
    address public owner;

    function transfer(address to, uint amount) public {
        require(msg.sender == owner, "Not owner");
        payable(to).transfer(amount);
    }
}

// Acceptable use: prevent contracts from calling
contract NoContractCalls {
    modifier noContracts() {
        require(tx.origin == msg.sender, "No contracts");
        _;
    }

    function sensitiveFunction() external noContracts {
        // Only EOAs can call
    }
}

🛡️ Prevention

  • Never use tx.origin for authentication
  • Always use msg.sender for access control
  • Only use tx.origin to check if caller is an EOA

Unsafe Delegatecall

External code modifies your storage

Critical

delegatecall executes code in the context of the calling contract, using the caller's storage. This can lead to storage corruption or complete contract takeover.

Real-world Impact: Parity Wallet (2017) — $150M frozen forever.
❌ VULNERABLE 
contract VulnerableProxy {
    address public owner;        // Slot 0
    address public implementation; // Slot 1

    // Anyone can call with arbitrary address!
    function execute(address target, bytes calldata data) external {
        target.delegatecall(data); // Attacker's code with OUR storage
    }
}

// Attacker deploys this and calls execute()
contract MaliciousImpl {
    address public owner; // Maps to slot 0 of proxy

    function attack() external {
        owner = msg.sender; // Overwrites proxy's owner!
    }
}

// Storage collision in proxy pattern
contract ImplementationV1 {
    uint256 public value; // Slot 0 - COLLIDES with proxy's owner!

    function setValue(uint256 _v) external {
        value = _v; // Overwrites proxy's storage!
    }
}
✅ SECURE - OpenZeppelin Proxy 
import "@openzeppelin/contracts/proxy/ERC1967/ERC1967Proxy.sol";
import "@openzeppelin/contracts-upgradeable/proxy/utils/UUPSUpgradeable.sol";

contract SecureImplementation is UUPSUpgradeable {
    // Storage starts AFTER proxy reserved slots
    uint256 public value;

    function _authorizeUpgrade(address newImpl) internal override {
        // Add access control here
    }
}

// Unstructured storage pattern avoids collisions
contract UnstructuredStorageProxy {
    // Random slot location
    bytes32 private constant IMPL_SLOT =
        bytes32(uint256(keccak256("eip1967.proxy.implementation")) - 1);

    function _getImplementation() internal view returns (address impl) {
        bytes32 slot = IMPL_SLOT;
        assembly { impl := sload(slot) }
    }
}

🛡️ Prevention

  • Never delegatecall to user-controlled addresses
  • Use established proxy patterns (ERC-1967, UUPS, Transparent)
  • Be extremely careful with storage layout in upgradeable contracts
  • Use OpenZeppelin's upgradeable contracts library

Front-Running & MEV

Transaction ordering exploitation

High

Attackers observe pending transactions in the mempool and submit their own with higher gas to execute first, extracting value through sandwich attacks and displacement.

❌ VULNERABLE 
contract VulnerableDEX {
    function swap(address tokenIn, address tokenOut, uint amountIn) external {
        // Attacker sees this in mempool
        // Front-runs with their own swap
        // Back-runs after victim's swap at worse price
        uint price = getPrice(tokenIn, tokenOut);
        // Execute swap...
    }
}

contract VulnerableAuction {
    function submitSolution(bytes32 answer) external {
        // Attacker sees the answer in mempool
        // Submits with higher gas to win
        require(keccak256(abi.encode(answer)) == puzzle);
        winner = msg.sender;
    }
}
✅ MITIGATED - Commit-Reveal 
contract SecureAuction {
    mapping(address => bytes32) public commits;
    uint256 public commitDeadline;
    uint256 public revealDeadline;

    // Phase 1: Commit (hidden)
    function commit(bytes32 hashedBid) external {
        require(block.timestamp < commitDeadline);
        commits[msg.sender] = hashedBid;
    }

    // Phase 2: Reveal (after commit phase)
    function reveal(uint256 bidAmount, bytes32 salt) external {
        require(block.timestamp >= commitDeadline);
        require(block.timestamp < revealDeadline);

        bytes32 hash = keccak256(abi.encodePacked(
            bidAmount, salt, msg.sender
        ));
        require(commits[msg.sender] == hash, "Invalid");
        // Process bid...
    }
}

// Slippage protection for DEX
contract ProtectedDEX {
    function swap(
        address tokenIn,
        address tokenOut,
        uint amountIn,
        uint minAmountOut,  // Slippage protection
        uint deadline       // Time protection
    ) external {
        require(block.timestamp <= deadline, "Expired");
        uint amountOut = calculateOutput(tokenIn, tokenOut, amountIn);

🛡️ Prevention

  • Use commit-reveal patterns for sensitive operations
  • Implement slippage protection and deadlines
  • Use batch auctions for fair ordering
  • Consider private mempools (Flashbots Protect)

Denial of Service (DoS)

Blocking legitimate users

High

Attackers prevent legitimate users from using the contract by causing functions to fail or consume excessive gas through unbounded loops or external call failures.

❌ VULNERABLE 
contract VulnerableAirdrop {
    address[] public recipients;

    function airdrop(uint amount) external {
        for (uint i = 0; i < recipients.length; i++) {
            // If ANY transfer fails, entire airdrop fails
            // Attacker adds contract that reverts in receive()
            payable(recipients[i]).transfer(amount);
        }
    }
}

contract VulnerableVoting {
    address[] public voters;

    function countVotes() external view returns (uint) {
        uint count = 0;
        // Gas limit exceeded if too many voters
        for (uint i = 0; i < voters.length; i++) {
            if (hasVoted[voters[i]]) count++;
        }
        return count;
    }
}
✅ SECURE - Pull Pattern 
contract SecureAirdrop {
    mapping(address => uint) public claimable;

    function setClaimable(address[] calldata recipients, uint amount) external {
        for (uint i = 0; i < recipients.length; i++) {
            claimable[recipients[i]] += amount;
        }
    }

    // Users pull their own funds
    function claim() external {
        uint amount = claimable[msg.sender];
        require(amount > 0, "Nothing to claim");
        claimable[msg.sender] = 0;
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success);
    }
}

contract SecureVoting {
    uint public lastProcessedIndex;

        if (end > voterCount) end = voterCount;

🛡️ Prevention

  • Use Pull over Push pattern for payments
  • Implement pagination for loops
  • Set maximum batch sizes
  • Avoid unbounded arrays in storage

Oracle & Price Manipulation

Flash loan price attacks

Critical

Attackers manipulate on-chain price data using flash loans to temporarily skew prices, enabling exploitation of DeFi lending and trading protocols.

Real-world Impact: Numerous DeFi hacks totaling hundreds of millions in losses.
❌ VULNERABLE - Spot Price 
contract VulnerableLending {
    IUniswapV2Pair public pair;

    function getPrice() public view returns (uint) {
        (uint reserve0, uint reserve1, ) = pair.getReserves();
        // Spot price can be manipulated with flash loans!
        return reserve0 * 1e18 / reserve1;
    }

    function borrow(uint collateral) external {
        uint price = getPrice(); // Manipulated!
        uint borrowLimit = collateral * price / 1e18;
        // Attacker deposits, manipulates price up,
        // borrows max, price returns, profit!
    }
}
✅ SECURE - Chainlink with Full Validation 
import "@chainlink/contracts/src/v0.8/interfaces/AggregatorV3Interface.sol";

contract SecureLending {
    AggregatorV3Interface internal priceFeed;
    // Set per-feed: ETH/USD heartbeat = 1h, some feeds = 24h
    uint256 public constant STALENESS = 1 hours;

    function getLatestPrice() public view returns (int256 price) {
        (
            uint80 roundId,
            int256 answer,
            ,
            uint256 updatedAt,
            uint80 answeredInRound
        ) = priceFeed.latestRoundData();

        require(answer > 0, "Oracle: non-positive price");
        require(updatedAt != 0, "Oracle: round not complete");
        require(block.timestamp - updatedAt <= STALENESS, "Oracle: stale price");
        require(answeredInRound >= roundId, "Oracle: stale round");

        return answer;
    }
}
✅ L2 Sequencer Uptime Check (Arbitrum/Optimism) 
// On L2s, the sequencer can go offline while Chainlink prices
// keep updating. Always verify sequencer is live before using prices.
contract L2SecureLending {
    AggregatorV3Interface internal priceFeed;
    // Chainlink Sequencer Uptime Feed address (per network)
    // Arbitrum One:  0xFdB631F5EE196F0ed6FAa767959853A9F217697D
    // Optimism:      0x371EAD81c9102C9BF4874A9075FFFf170F2Ee389
    AggregatorV3Interface internal sequencerFeed;

    uint256 public constant GRACE_PERIOD = 1 hours;
    uint256 public constant STALENESS    = 1 hours;

    function _checkSequencer() internal view {
        (, int256 answer, uint256 startedAt,,) = sequencerFeed.latestRoundData();
        // answer == 0: sequencer is UP; answer == 1: sequencer is DOWN
        require(answer == 0, "Oracle: L2 sequencer down");
        // Reject prices from right after sequencer restart
        require(block.timestamp - startedAt > GRACE_PERIOD, "Oracle: grace period active");
    }

    function getLatestPrice() external view returns (int256) {
        _checkSequencer();

        (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
        require(price > 0, "Oracle: non-positive price");
        require(block.timestamp - updatedAt <= STALENESS, "Oracle: stale price");

        return price;
    }
}

🛡️ Prevention

  • Never use spot prices for critical operations
  • Validate: price > 0, updatedAt != 0, staleness, answeredInRound >= roundId
  • On L2 (Arbitrum/Optimism): always check the Chainlink Sequencer Uptime Feed
  • Enforce a grace period after sequencer restarts (min 1 hour)
  • Set STALENESS per-feed — heartbeats vary (1h for ETH/USD, up to 24h for others)
  • Implement circuit breakers for extreme price movements

Signature Replay Attacks

Reusing valid signatures maliciously

High

Valid signatures can be reused maliciously on the same contract, different chains, or different contracts if proper replay protection is not implemented.

❌ VULNERABLE 
contract VulnerablePermit {
    function permitTransfer(
        address from,
        address to,
        uint amount,
        bytes memory signature
    ) external {
        bytes32 hash = keccak256(abi.encodePacked(from, to, amount));
        address signer = recoverSigner(hash, signature);
        require(signer == from, "Invalid");

        // Same signature can be used multiple times!
        // And on other chains where contract is deployed!
        balances[from] -= amount;
        balances[to] += amount;
    }
}
✅ SECURE - EIP-712 with Nonce 
contract SecurePermit {
    mapping(address => uint) public nonces;
    bytes32 public immutable DOMAIN_SEPARATOR;

    bytes32 public constant PERMIT_TYPEHASH = keccak256(
        "Permit(address from,address to,uint256 amount,uint256 nonce,uint256 deadline)"
    );

    constructor() {
        DOMAIN_SEPARATOR = keccak256(abi.encode(
            keccak256("EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"),
            keccak256(bytes("SecurePermit")),
            keccak256(bytes("1")),
            block.chainid,      // Chain protection
            address(this)       // Contract protection
        ));
    }

    function permitTransfer(
        address from, address to, uint amount,
        uint deadline, uint8 v, bytes32 r, bytes32 s

Division Before Multiplication

High

Solidity integer division truncates toward zero. Dividing before multiplying amplifies truncation error — in DeFi at scale, this creates exploitable rounding that extracts value from the protocol or from users. Always multiply before you divide.

🗄️ Storage Layout & Upgrade Safety

Storage slot collisions and layout corruption are silent killers in upgradeable contracts.

Storage Collision & Layout Corruption

Variables overwrite each other across upgrades or proxy boundaries

High

Upgradeable contracts share storage between proxy and implementation. Adding, removing, or reordering state variables in an upgrade corrupts all existing data. Even inheritance order changes can shift slot assignments. ERC-7201 namespaced storage eliminates this class of bug entirely.

Root Cause: Solidity assigns storage slots sequentially (slot 0, 1, 2…). Any change to variable order or count between upgrades misaligns slots in the live proxy. With delegatecall, proxy admin variables can also collide with implementation storage at the same slots.

The Curve Finance exploit (August 2023, ~$73M) was caused by a reentrancy guard failure in Vyper versions 0.2.15, 0.3.0, and 0.3.1. Separate compiler bugs in older versions affect storage array bounds checking. Always verify which Vyper version a contract was compiled with before auditing.

Compiler version matters: Bug 1 (reentrancy guard) — fixed in 0.3.2+. Bug 2 (array bounds) — fixed in 0.3.10. Default external visibility is not a bug but a language design difference that surprises Solidity developers.
❌ VULNERABLE — Vyper ≤0.3.1 @nonreentrant bug 
# Compiled with Vyper 0.3.0 or 0.3.1
# @nonreentrant guard is BROKEN — mutex write happens after the call

interface Pool:
    def remove_liquidity() -> uint256: nonpayable

@external
@nonreentrant("lock")   # guard fails in these versions
def withdraw(amount: uint256):
    # Attacker can re-enter here before lock is set
    Pool(self.pool).remove_liquidity()
    self.balances[msg.sender] -= amount
✅ SAFE — Vyper 0.3.10+ with CEI pattern 
# Use Vyper 0.3.10+ (check: vyper --version)
# Defense-in-depth: apply CEI regardless of @nonreentrant

@external
@nonreentrant("lock")   # fixed in 0.3.10+
def withdraw(amount: uint256):
    # Checks
    assert self.balances[msg.sender] >= amount
    # Effects — state updated BEFORE external call
    self.balances[msg.sender] -= amount
    # Interactions
    send(msg.sender, amount)
❌ VULNERABLE — DynArray bounds bypass (Vyper <0.3.10) 
# Vyper <0.3.10: bounds check uses MAX_SIZE, not len(arr) at runtime
# DynArray[address, 500] — 500 slots allocated, only 5 stored
whitelist: DynArray[address, 500]
owner: address                        # storage slot AFTER whitelist block

@external
def add(addr: address):
    self.whitelist.append(addr)       # stores at indices 0..4 (len=5)

@external
@view
def get(idx: uint256) -> address:
    return self.whitelist[idx]        # SHOULD revert at idx>=5, but doesn't!
                                      # idx=5..499 reads uninitialised storage
                                      # idx overlapping owner slot leaks self.owner

@external
def corrupt(idx: uint256, addr: address):
    self.whitelist[idx] = addr        # WRITE to arbitrary storage slot
                                      # self.whitelist[499] may overwrite self.owner
✅ SAFE — explicit bounds guard (all Vyper versions) 
# Defense: always assert idx < len(arr) before indexed access
# Safe regardless of Vyper version — never rely on compiler bounds alone

@external
@view
def get(idx: uint256) -> address:
    assert idx < len(self.whitelist), "out of bounds"
    return self.whitelist[idx]

@external
def update(idx: uint256, addr: address):
    assert idx < len(self.whitelist), "out of bounds"
    self.whitelist[idx] = addr

# If on Vyper >=0.3.10 the compiler also checks, but explicit asserts
# give better error messages and survive future refactors
❌ Visibility trap — function exposed externally 
# In Vyper, all functions are external by default
# No "private" or "internal" visibility unless explicitly declared

def _compute_fee(amount: uint256) -> uint256:   # exposed externally!
    return amount * self.fee_rate / 10000

# Anyone can call _compute_fee() directly
✅ Use @internal decorator explicitly 
@internal
def _compute_fee(amount: uint256) -> uint256:   # truly internal
    return amount * self.fee_rate / 10000

@external
def swap(amount: uint256) -> uint256:
    fee: uint256 = self._compute_fee(amount)    # self. prefix required
    return amount - fee

🛡️ Prevention

  • Check compiler version: vyper --version — avoid 0.2.15, 0.3.0, 0.3.1 for contracts using @nonreentrant
  • Apply CEI pattern as defense-in-depth regardless of @nonreentrant
  • Mark all internal helpers @internal — don't rely on naming convention (_prefix)
  • Use Caracal (Trail of Bits) for Vyper static analysis
  • Check Vyper security advisories for the exact compiler version before auditing

🛡️ Security Patterns

Battle-tested patterns to secure your smart contracts.

Checks-Effects-Interactions

Validate inputs, update state, then make external calls. Prevents reentrancy and ensures consistent state.

Pull over Push

Let users withdraw funds instead of pushing payments. Prevents DoS from failing transfers.

Emergency Stop

Circuit breaker pattern with Pausable. Allows halting operations during incidents.

Rate Limiting

Cooldown periods between actions. Limits damage from compromised accounts.

Commit-Reveal

Two-phase submission hides data until reveal. Prevents front-running.

Guard Check

Modifiers for common validations. Clean, reusable input validation.

Combined Pattern Example 
import "@openzeppelin/contracts/utils/ReentrancyGuard.sol";
import "@openzeppelin/contracts/utils/Pausable.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract SecureVault is ReentrancyGuard, Pausable, Ownable {
    mapping(address => uint) public balances;
    mapping(address => uint) public lastAction;
    uint public constant COOLDOWN = 1 hours;

    error ZeroAmount();
    error Cooldown();

    modifier rateLimit() {
        if (block.timestamp < lastAction[msg.sender] + COOLDOWN)
            revert Cooldown();
        lastAction[msg.sender] = block.timestamp;
        _;
    }

    function withdraw(uint amount)
        external
        nonReentrant
        whenNotPaused
        rateLimit
    {
        // Checks
        if (amount == 0) revert ZeroAmount();
        require(balances[msg.sender] >= amount);

        // Effects
        balances[msg.sender] -= amount;

        // Interactions
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success);
    }

    function pause() external onlyOwner { _pause(); }
    function unpause() external onlyOwner { _unpause(); }
}

⛽ Gas Optimization

Techniques to reduce transaction costs without sacrificing security.

Storage & Loop Patterns 
// ✅ calldata for read-only params (cheaper than memory)
function process(uint256[] calldata data) external pure returns (uint256 sum) {
    uint256 len = data.length; // cache .length — SLOAD is 2100 gas first access
    for (uint256 i; i < len;) {
        sum += data[i];
        unchecked { ++i; } // safe: can't overflow before gas limit
    }
}

// ✅ Storage packing — pack < 32-byte types into same slot
// ❌ Bad: 3 slots (uint8 doesn't share with adjacent uint256)
struct BadConfig {
    uint256 a;   // slot 0
    uint8   b;   // slot 1
    uint256 c;   // slot 2
}
// ✅ Good: 2 slots
struct GoodConfig {
    uint256 a;   // slot 0
    uint256 c;   // slot 1
    uint8   b;   // slot 1 (packed alongside c's remaining space — if c < 32 bytes)
}
// Tip: sort struct fields largest→smallest to maximize packing

// ✅ Manual transient lock (Solidity assembly)
contract ManualTransientLock {
    uint256 private constant LOCK_SLOT = uint256(keccak256("reentrancy.lock")) - 1;

    modifier transientLock() {
        assembly { if tload(LOCK_SLOT) { revert(0, 0) } }
        assembly { tstore(LOCK_SLOT, 1) }
        _;
        assembly { tstore(LOCK_SLOT, 0) }
    }
}

// ✅ Transient storage for intra-tx accumulators (flash loan fees, etc.)
// Accumulate a value across multiple calls in one tx, read at end
contract FlashLoanFeeAccumulator {
    uint256 private constant FEE_SLOT = uint256(keccak256("flash.fee.accumulator")) - 1;

    function _addFee(uint256 fee) internal {
        assembly {
            let current := tload(FEE_SLOT)
            tstore(FEE_SLOT, add(current, fee))
        }
    }
    function _getTotalFee() internal view returns (uint256 fee) {
        assembly { fee := tload(FEE_SLOT) }
    }
}

// ⚠️ Security note: transient storage persists within the SAME tx
// A reentrancy attack WITHIN the same tx can still read/write tstore
// → still need locks; transient just makes them cheaper, not unnecessary

🏦 DeFi Audit Checklists

Protocol-specific security checks. Click items to track your progress.

Pool Logic
  • Pool invariants (x*y=k) correctly maintained after every operation
  • Pool reserves not manipulable via direct token transfers
  • Fee calculation correct and not gameable through rounding
Slippage & MEV
  • All swaps require amountOutMin — never accepts 0
  • Deadlines enforced on all swap transactions
  • TWAP used for pricing — not spot price from getReserves()
  • First deposit inflation attack prevented (minimum liquidity / dead shares)
LP Token Accounting
  • LP minting is proportional to liquidity provided
  • Reentrancy guard on all state-changing pool functions
  • Fee-on-transfer tokens handled with balance-delta measurement
Oracle & Pricing
  • No spot price oracles — use TWAP or Chainlink
  • Chainlink staleness check: updatedAt + answer > 0 + sequencer uptime (L2)
  • Fallback if primary oracle fails
Collateral & Liquidation
  • Collateral factor changes have delay — no instant liquidation of existing positions
  • Liquidation threshold correctly above collateral factor
  • Partial liquidations do not leave dust positions
  • Bad debt socialization mechanism defined
  • Cannot borrow own collateral (recursive self-collateralization)
  • Health factor re-checked after every action that changes it
Share Accounting
  • First depositor inflation attack prevented (virtual offset or dead shares)
  • Rounding direction always favors the vault (not the user)
  • convertToShares and convertToAssets are consistent
  • previewDeposit matches actual deposit execution
  • Share price can only decrease via actual losses
Fees & Assets
  • totalAssets() includes all protocol assets including deployed strategy funds
  • Fees cannot be extracted retroactively from existing depositors
  • Maximum fee hard-cap — admin cannot set 100% fee
  • Emergency strategy withdrawal path exists
Flash Loan Resistance
  • Voting uses getPastVotes(voter, snapshotBlock) — not live balance
  • Voting delay between proposal creation and vote start
  • Quorum not trivially achievable by single entity
Proposals & Execution
  • Minimum proposal threshold prevents spam
  • Timelock between approval and execution (min 48h)
  • Executed proposals marked — cannot re-execute
  • Emergency bypass appropriately restricted and documented
Message Validation
  • Messages signed by sufficient validators (check threshold, not just ≥1)
  • Replay protection: nonce per chain + chain ID in message hash
  • Source chain identity validated on destination
  • Message timeouts enforced
Token & Validator
  • Lock-and-mint accounting balances correctly across chains
  • No single entity controls majority of validators
  • Slashing mechanism exists for malicious validators
  • Failed destination tx has retry or refund path
